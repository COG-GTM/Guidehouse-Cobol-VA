Modernization improvement findings — leap-year validator

The COBOL ⇔ Python runtime parity loop exposed exactly one byte-for-byte divergence. It is not a regression — it is a corrected, 126-year-old calendar bug in the legacy code. This page is the SME-facing record of the divergence, classified and put in front of you for a yes/no decision.

What we found

The legacy COBOL date-validation routine (DATECONV.cbl, paragraph 9950-VALIDATE-YYYY) uses the old Julian leap-year rule: every year divisible by 4 is a leap year, no exceptions. The modernized Python port uses the standard proleptic Gregorian rule:

A year is a leap year if divisible by 4, except centuries are not leap years unless also divisible by 400.

The two rules agree on every year except century years not divisible by 400 (1700, 1800, 1900, 2100, 2200, 2300, …). For all other years in scope of this system the two implementations are byte-for-byte identical, which is why 79 of 80 test vectors match.

Future-hazard note. 1900 is the historical case visible in the parity test today, but the same defect causes the legacy validator to accept three more invalid dates this century and beyond — 02/29/2100, 02/29/2200, 02/29/2300. Any downstream system that uses this validator past 2099 would silently admit a date that does not exist on the Gregorian calendar.

Why we flagged it instead of silently mirroring the legacy

A naive port that prioritizes byte-for-byte legacy fidelity above all else would have been instructed to backport the Julian rule into the Python validator so the parity diff comes out clean. We deliberately did not do that. Two reasons:

  • Mathematically incorrect dates should not enter the modernized system. The legacy was wrong; quietly carrying the bug forward into the modernized stack inherits a known, future-impacting defect.
  • Silent mirrors are unauditable. If we had just copied the legacy behavior with no callout, the SME would have no way to know we made a calendar-correctness trade-off on their behalf. Surfacing it explicitly — with the exact COBOL paragraph cited, the test vector visible, and a classification of modernization_improvement rather than unresolved_mismatch — preserves their right to decide.

This is the answer to the "are you sure you're not silently regressing the legacy?" question that federal acquisition always asks. We caught the divergence, we classified it correctly, and we put it in front of you.

SME decision

The modernized stack ships today with Option A. Switching to Option B is a one-line change in migration/converted-code/python/dateconv.py; the parity diff would then go to 80/80 byte-for-byte but the modernized system would re-inherit the 1900 / 2100 / 2200 / 2300 bug.

Recommended · default · shipped

Option A — Keep the modernization fix

The Python port uses proleptic Gregorian. 02/29/1900 is rejected. Future century-year edges (2100/2200/2300) are also rejected. Parity diff stays at 79/80 with the 1 mismatch classified as a documented improvement.
