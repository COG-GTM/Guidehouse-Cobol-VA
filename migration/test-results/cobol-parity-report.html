What the runtime parity loop caught

Before the GnuCOBOL harness existed, the Python port of DATECONV had 13 defects that all 77 unit tests passed cleanly. Compiling the customer's source/cobol/DATECONV.cbl verbatim under GnuCOBOL 3.1.2 and diffing byte-for-byte against the Python port over 52 vectors exposed them. They were patched in the same commit as the harness; this report regenerates daily.

Why this matters for federal buyers. Modernization efforts routinely ship "semantically equivalent" ports that pass their own unit tests and then drift from the legacy system in edge cases the test suite never covered. A runtime parity loop — compile the customer's actual COBOL, diff byte-for-byte — is the verification federal acquisition wants. This harness caught 13 such defects on the first run and now executes on every push to the branch.

Methodology

  • GnuCOBOL build: cobc (GnuCOBOL) 3.1.2.0.
  • COBOL compile path: Direct compile of source/cobol/DATECONV.cbl succeeded under GnuCOBOL (no preprocessor required; no source modifications)..
  • Python port: migration/converted-code/python/dateconv.py — DateConv class with 40 _runNN_* methods mapped 1:1 to DATECONV.cbl paragraphs, dispatched via DATESUB-FUNC..
  • Driver: migration/test-results/build/DATECONV-DRIVER.cob reads pipe-delimited vectors from STDIN, populates the LINKAGE-equivalent CONV-DATES record exactly as defined in source/copybooks/DATECONV-WS.cpy, CALLs the customer subprogram DATECONV, and writes a 12-column result row per vector. See EMIT-RESULT for the exact PIC clauses used to render the output.
  • Python adapter: migration/test-results/run_cobol_parity.py parses the same vector format, builds a ConvDates dataclass, calls DateConv().dispatch(func_code, cd), and renders the result with the same PIC clauses (zero-padded numerics, signed PIC -(5)9 for DAYS-DIF, two-digit status code).
  • Diff: every output column is compared as a string. A row is "matched" only if all 12 columns are byte-identical.

Honesty / Caveats

GnuCOBOL is an open-source COBOL-85+ implementation, not the customer's Unisys runtime. The customer's source/cobol/DATECONV.cbl compiled directly under GnuCOBOL without any preprocessor or source modification — the MIGRTN comment-column markers and the word/alignd annotations in the JDN copybooks live in the sequence area (columns 1–6) and are ignored by fixed-format COBOL, so no Unisys-specific directives needed to be stripped. Compiler warnings (OCCURS DEPENDING ON truncation, redefines size mismatches) were observed but did not affect the four COBOL-85 intrinsic functions that DATECONV ultimately calls.

The four intrinsics actually exercised — INTEGER-OF-DATE, DATE-OF-INTEGER, INTEGER-OF-DAY, DAY-OF-INTEGER — are standardised in COBOL-85 / FIPS 21-3 and behave identically across conformant compilers. Differences vs. the Unisys runtime would only appear in implementation-defined corners (e.g. truncation semantics for arithmetic overflow). The customer's JDN-RECORD-ACCESS.cpy states explicitly that "the COBOL-85 standard (FIPS 21-2) was expanded in 1989 to include the Intrinsic Functions Module ... the date routines it contains are a FIPS standard that the TST can rely on to be implemented on all future COBOLs", which is exactly the portability claim GnuCOBOL satisfies.

The harness does not mutate source/cobol/DATECONV.cbl or the JDN copybooks. The only generated artifact under migration/test-results/build/ is the COBOL driver (DATECONV-DRIVER.cob) used to bridge stdin/stdout into the subprogram's CALL interface.

These vectors diverge from the legacy COBOL by design. The Python port has been documented in analysis/dateconv-function-inventory.md and the customer parity row is set to "deliberate improvement" rather than "regression". They do not affect the exit status of the harness.
